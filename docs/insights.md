# Insights

Workflow and tooling observations for future agent sessions. Keep this file free of feature-specific implementation details.

- The GitNexus index may lag behind worktree commits even when it does not emit a staleness warning. Use it for orientation, then verify current behavior by reading source files directly.
- PowerShell treats square brackets in paths as wildcard syntax. Use `Get-Content -LiteralPath` for files under dynamic route folders such as `app\runs\[id]\page.tsx`.
- `.github/git_notes_template.md` may be absent in early scaffold worktrees. When a task still requires a git note, attach a concise manual note with summary, validation, and concerns.
- `Get-ChildItem -Recurse .github` can wander into dependency package metadata when the root `.github` directory is absent. Prefer `Test-Path -LiteralPath .github` before recursive listing.
- Initial `npm install` in OneDrive-backed worktrees may time out and leave a partial `node_modules`; verify `package-lock.json` exists before trusting the install, and delete only the resolved local dependency tree before retrying.
- `gh pr create --json` is not supported by the installed GitHub CLI version. Use `gh pr create` first, then `gh pr view --json ...` to capture structured PR metadata.
- Grok review sessions may create untracked `mcps/` descriptor files and emit unrelated Vercel MCP auth warnings. Delete generated `mcps/` artifacts before committing and treat the auth warnings as review-tool noise unless the task uses Vercel.
- `next lint` currently works after adding ESLint config but prints a deprecation warning. Future sessions should consider migrating the lint script to the ESLint CLI.
- Grok-as-implementer (Step 4) executes well *within* a task's scope (correct TDD, files, code matching the contract) but grades itself only against the test files named in its prompt. In this session it reported "25 tests pass" for a runner change that had broken the SSE route + 2 other tests; only the main agent's **full** `npm test` caught it. Always run the full suite + `tsc` + lint at the orchestrator level after each delegated task — never trust the scoped self-report.
- Grok consistently omits the trailing newline on every file it writes/edits, regardless of explicit instructions. Normalize newlines during review (`printf '\n' >> file` where `tail -c1` is non-empty) rather than relying on the prompt.
- The expensive part of planning is the cross-task contract (decomposition + per-task Interfaces + build order + "who else calls this signature"), not the verbatim per-step code. A narrowly-scoped implementer structurally cannot derive that contract. Keep a **lightweight** plan (contract skeleton) and let the implementer regenerate code/commands — this retains the spec→plan→implementation verification chain at lower cost.
- The `next/no-assign-module-variable` ESLint rule rejects a local variable named `module`; the gated/typecheck steps pass but lint fails. Avoid `module` as an identifier (use `moduleName` etc.).
- **OneDrive corrupts git worktree metadata.** A worktree under the OneDrive-synced project tree had its `.git/worktrees/<name>/` partially synced — `HEAD`, `commondir`, `gitdir`, and `index` were missing (only empty `logs/`/`refs/` survived), so every `git` command in it failed with `fatal: not a git repository` and `git worktree list` silently dropped it. The working files were intact; only git metadata was destroyed. Recovery: from the main repo, `git worktree prune` then create a fresh worktree from `main` and redo any uncommitted work there. The corrupt admin dir may refuse deletion (`Permission denied`) while files are locked — harmless once de-registered. Consider placing `.worktree/` outside the OneDrive-synced path to avoid recurrence. After resuming in any worktree, `git status` (Workflow Rule 8) catches this immediately.
- **Tests must live under `tests/`, not co-located with source.** `vitest.config.ts` includes only `tests/**/*.test.ts`. A plan that places a unit test at `lib/**/*.test.ts` will pass when run directly (`npx vitest run <path>`) but is silently excluded from `npm test` — so the orchestrator's full-suite gate never runs it. Specify the `tests/<name>.test.ts` path in the lightweight plan, and sanity-check that new test counts actually rise in the full-suite output.
- The Bash-tool shell cwd resets to the invoking (here: broken) worktree after every call; shell state does not persist. When operating on a different worktree, prefix every command with `cd "<worktree>" &&` or use `git -C "<worktree>"` / absolute paths — do not assume a prior `cd` carried over.
- Capturing Grok handoff output as `grok -p … --output-format json > file.json` and parsing `text`/`sessionId` with a one-line `node -e` was reliable; foreground calls with a high timeout finished in ~1–3 min per task. Newline omission was intermittent this session (some files clean, some not) — checking `tail -c1` per touched file remains cheaper than predicting.

## Grok Build Implementation/Review Handoff — Phase 1 retrospective

- **Net assessment:** the handoff delivered real gains. Across 9 implementation tasks + 2 fix batches + a security review + a bundled code review, Grok was accurate *within scope* every time and produced one genuinely high-value artifact (the code review). The division of labor — main agent owns the cross-task contract + verification, Grok owns implementation + review — held up. Overhead (newline fixes, artifact cleanup, session deletion, full-suite re-validation) was real but modest against the gains.
- **Biggest concrete improvement for the handoff prompt:** instruct the implementer to run the **full** `npm test` + `npx tsc --noEmit` as its OWN self-check, not just the test files named in its task. Grok *can* run the whole suite when told; scoping its self-check to named files is exactly what hid the Task 5 SSE-route breakage (see the scoped-self-report entry above). Moving the full-suite run into the delegated prompt catches cross-task breakage *before* the round-trip back, instead of only at the orchestrator's post-task gate. The orchestrator gate stays mandatory regardless.
- **Grok-as-reviewer is the standout win.** The `/bundled:review` delegation surfaced 15 findings, ~7 real bugs the spec/plan/implementation all missed — including cross-cutting ones a scoped implementer cannot see (a workflow still emitting prose, an orphaned-but-broken server action, a silent rerun-failure path). The `/security-review` delegation correctly distinguished intended sandbox execution from host vulnerabilities (no false positives on the by-design RCE-in-sandbox). Treat review delegation as a primary value source, not a formality.
- **Review-subagent environment is fragile.** The bundled reviewer hit `Tool not found: Write/Shell/StrReplace` and `tool_output_error` on `Read`, and `gh` diff output carried ANSI color codes that broke inline-comment line matching until re-fetched clean. It still posted the PENDING review via workarounds, but: prefer `NO_COLOR=1` / `gh ... --json` when feeding `gh` output to the reviewer, and expect the skill to fall back to returning the review body in `text` when it cannot write files.
- **Tool-artifact dirs to ignore proactively:** Grok/tooling runs drop `mcps/`, `terminals/`, and `agent-tools/` (the last holding large review-diff dumps) into the cwd. Keep these in `.gitignore` from the start so `git status` stays clean for the per-task commit gate.
- **Trailing-newline omission is intermittent, not constant** (contrary to the earlier absolute phrasing): later batches landed clean newlines while early tasks did not. Still cheaper to always check (`tail -c1`) and normalize than to predict.
